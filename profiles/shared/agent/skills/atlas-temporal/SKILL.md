# Atlas (Temporal) at Datadog

## Overview

Atlas is Datadog's workflow platform built on Temporal. It provides:
- **Atlas Server** (Temporal server + worker framework)
- **Atlas workers** (team-owned services running workflows/activities)
- **Protobuf SDK** (type-safe workflow/activity APIs over Temporal)
- **Tools**: Atlas CLI, Atlas UI, Temporal UI

Use this skill to answer Atlas/Temporal questions by grounding responses in the repo and ATLAS Confluence docs. Always prefer the latest Confluence content for operational details.

## Core Concepts to Communicate

### Atlas Platform Components
- **Temporal Server**: open-source orchestration engine.
- **Protobuf SDK**: type-safe interface for workflows/activities; reduces boilerplate.
- **Worker Framework**: shared tooling around deploying/observing workers.

### Protobuf SDK (high-level)
- Workflows/activities are defined in `.proto` files.
- Code is generated by `protoc-gen-temporal-go` and `protoc-gen-temporal-testhelper`.
- Main generated artifacts:
  - `<worker>.temporal_client.pb.go`: start workflows from external callers.
  - `<worker>.temporal_worker_client.pb.go`: call workflows/activities within workflows.
  - `<worker>.temporal_worker.pb.go`: interface your worker implements.
  - `<worker>.temporal_test.pb.go`: test helpers.
- Benefits: type safety, discoverability, multi-language, reduced boilerplate, safer encoding.

### Atlas CLI and UI
- **Atlas CLI** is the primary tool to start/signal/query workflows.
- **Atlas UI** provides a browser-based workflow starter; if unavailable, use Atlas CLI.

### Atlaslite
- Datadog's single-process Temporal distribution (based on Temporalite).
- Intended for local use or lightweight deployments; see `apps/atlaslite` docs.

### Rapid-specific Atlas guidance
- **Rapid apps live in `apps/`**: Rapid doesn't support `internal/` yet, so Atlas Rapid services remain under `domains/atlas/apps/*`.
- **Workflow Starter API** is a Rapid HTTP API backing the Atlas UI legacy workflow starter. Local dev uses `rapid run --domain=atlas --service=workflow-starter` and internal auth tokens from `ddtool auth token atlas --datacenter <dc>`.
- **Rapid / gRPC integration**: enable `(temporal.experimental_grpc_proxy_config)` in your service proto, add `go_atlas` to `# gazelle:dd_proto_lang`, then run `bzl run //:gazelle` and `bzl run //:snapshot -- //path/...`. Register the generated `*AtlasProxy` in your worker.
- **Local Rapid worker testing**: use `domains/atlas/etc/worker-local-scripts` with `--rapid` and start a local server via `rapid go-atlas start-server`.
- **RMP deployments**: Rapid framework doesn't support RMP, so some APIs use custom Helm charts + manual image/chart replication (`apps/apis/*/config/k8s/README.md`).
- **Bootstrap templates** for workers are shared by Atlas CLI and Rapid CLI (`domains/atlas/shared/libs/bootstrap`).
- **TemporalAuthMiddleware** for Rapid APIs is deprecated; consult `#atlas` before using (`domains/atlas/shared/libs/rapid/middleware.go`).

### SLOs (from Service Overview)
- Critical RPCs (start/signal/cancel/terminate) drive SLOs.
- Availability SLO: 99.9% of critical RPCs successful.
- Latency SLO: p99 critical RPCs within 200ms (99% of time).

## Worker Deployment Guidance (Quick Checklist)

Use the **Troubleshooting worker deployment issues** page for details. Common checks:
- Validate deploy config: `ddr deploy validate configs path/to/deploy.yaml --with-values-js`.
- Validate service catalog: `bzl run //domains/coreobs/apps/servicecatalog/cmd:service-catalog -- validate-yaml path/to/service.datadog.yaml`.
- Namespace uses dashes (no underscores).
- `workday_team` matches Workday org chart exactly.
- If deploying outside ddbuild, ensure registry replication settings + `release_target`.

## Graceful Shutdown

- Default worker graceful shutdown: 60s.
- Configurable via Helm values (`temporal.gracefulShutdownTimeoutSeconds`).
- Kubernetes pod termination timeout is 5 minutes max; worker shutdown must be <= 5m.
